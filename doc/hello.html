Vision

Key Goals

  • Enable rapid development of RPC services
  • Expose those services on the network with ease
  • Single service implementation, many transports/protocols
  • Fast client that supports all transports/protocols

History

  • Grew out of frustration with LShift's JSON-RPC toolkit
  • No validation
  • HTTP only
  • JSON-RPC 1.0 only
  • All requests go through single gen_server

3/11

Architecture (RPC Handlers)

  • Handler modules expose service metadata through callbacks

Stateless Handlers (hello_stateless_handler behaviour)

  • Requests are independent of each other and may execute concurrently
  • Does not maintain peer session
  • No asynchronous replies, no out-of-band notifications

Stateful handlers (hello_stateful_handler behaviour)

  • Keeps session state between requests
  • One session per peer ("peer" is transport-dependent)
  • Handler process may receive Erlang messages and send notifications to the client.
  • API feels like gen_server

Architecture (RPC Clients)

hello_client

  • API similar to database clients
  • Create a client, then execute calls on it
  • Support for server-side notifications
  • Implemented as gen_server

hello_simple_client

  • For testing and debugging only
  • Starts a new client process each time
  • No notifications

Architecture (Transports)

  • ZeroMQ transport uses erlzmq2
  • HTTP transport uses cowboy for server, ibrowse as client

Server transports (hello_binding behaviour)

  • Construct supervisor child specification for listener
  • Generate registry key for listener
  • Push binary messages through hello_binding API
  • Receive binary reply messages asynchronously

Client transports (hello_client behaviour)

  • Transport runs in the generic client process
  • Get request records through send_request/3 callback
  • Handle replies through handle_info/3 callback
